Act 4 - Behind the scenes

Anatomy of a commentable file.

One file, four stable ownership lanes

  • CSS region theme and layout rules the build can replace in one shot.
  • COMMENT UI region toolbar, sidebar, menus, and export chrome owned by the layer.
  • JS region runtime behavior, anchoring, exports, and widget logic.
  • CONTENT region your slides, report prose, diagrams, and examples - the part the author owns.

That separation is why upgrades stay deterministic: swap CSS, COMMENT UI, and JS, leave CONTENT alone, then re-stamp the version.

<main data-cmh-content-root data-cmh-mode="deck">
  <!-- BEGIN: CSS -->
  <!-- BEGIN: COMMENT UI -->
  <!-- BEGIN: JS -->
  <!-- BEGIN: CONTENT -->
  <section class="slide"> ... </section>
  <!-- END: CONTENT -->
</main>

The build swaps only the layer-owned regions and re-stamps the version. Your content is never clobbered.

Act 4 - Anchoring

How a comment finds the same spot on reload.

Text offsets

Prose and cells anchor by quote plus character range, resilient to small edits around them.

Structural keys

Charts, images, Mermaid nodes, and slides anchor by a stable element key, not a pixel position.

Widget deltas

Board moves and checklist toggles record the before and after state as part of the bundle.

Quote: "Bed 4 / Squash / 44 L"
Pinpoint: Act 2 > Evidence > watering table, row 2
Stable id: slide-d7119544::table-bed-4-liters
Note: make the high-water row easier to justify
HANDLED_IDS_JSON: ["cmh-bed4-liters"]

Stable ids let the agent mark a comment handled; on reload, handled-comment pruning clears it while unresolved notes stay put.

Act 4 - Portability

Three portability modes explain every handoff.

Mode Best for Styles + runtime Mermaid + charts Comments
Non-portable Fast local iteration skill folder CDN browser storage
Portable Self-contained review or archival inlined CDN seeded from HTML, then browser storage
Offline Zero-network handoff inlined vendored runtimes seeded from HTML, then browser storage

CDN means library code only. Your document text and comments stay local.

Act 4 - Safe by construction

How the skill is built.

Main skill file

SKILL.md is the entry point: when to use Commentable HTML, which authoring path to take, and which final validation steps are mandatory.

On-demand references

references/document-layout.md and references/design-decisions.md stay loaded on demand to keep context minimal until the agent needs that exact contract.

Deterministic editors

tools/authoring/new_document.py, tools/authoring/retrofit.py, and tools/authoring/upgrade.py do surgical, repeatable edits instead of freehand rewriting.

Mandatory final gate

tools/validate/validate.py --strict is the last stop before HTML goes back to the user, so malformed structure never slips through.

Act 4 - Validation methodology

Testing and validation keep the HTML honest.

UX-first browser checks

Playwright opens the real demos, clicks, drags, exports, and reloads them the way a reviewer does. plugin-tests.yml reruns that browser suite in CI.

Deterministic Python coverage

Python test_*.py files pin the builders, validators, and deck scaffolds so structure or safety regressions fail before publish.

Cross-platform and cross-agent

The repo reruns validators on Windows, macOS, and Linux, and the same portable HTML works in Copilot and Claude because the output is just HTML.

Anti-regression by default

Every bug fix or new behavior adds a spec row and a covering test, so the next rebuild proves the contract still holds.
