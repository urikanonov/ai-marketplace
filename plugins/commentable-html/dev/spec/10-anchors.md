## Layered checklist

A `data-cmh-checklist` container is a hierarchical checklist. Any descendant carrying `data-cmh-state` or `data-cmh-item` is an item; an item with child items (nested lists, or table rows that name a parent via `data-cmh-parent`) is a branch that aggregates over its DIRECT children, otherwise a leaf. Leaf state cycles `blank -> check -> cross -> question -> blank`; a branch click propagates its next state to every descendant leaf. Only leaves whose state differs from their authored `data-cmh-state` baseline are stored, as one-character codes under `COMMENT_KEY + "::cl"`. Changes surface as one per-list card (jump + reset) placed by document order, a Copy-all section with `CHECKLIST_STATE_JSON`, and an export bake into `data-cmh-state`. Item identity is `data-cmh-item` when present, else the 1-based document-order position within the checklist; identity is never rendered (no visible numeration), so a checklist can live in a sortable table.

## Editable notes fields

A `data-cmh-note="<id>"` element is a single editable free-text field. The runtime replaces its
authored text with a `<textarea>` (with a single/multi-line toggle) whose baseline is the authored,
canonically-normalized `textContent`. Only a note whose current text differs from baseline is stored,
as `{id:text}` under `COMMENT_KEY + "::note"`; edits surface as one per-note change card (jump + reset)
placed by document order, a Copy-all section with `NOTES_STATE_JSON`, and an export bake into the
element's text. The field is `cm-skip` (editing never creates a highlight) and is set up before offset
restoration so its excluded text does not shift comment offsets. The one canonical normalizer (LF line
endings, trimmed outer whitespace) is implemented identically in the runtime and in
`tools/notes/notes_apply.py` so the browser and the cementing tool agree.
