## Section review tracking

| Feature id | Behavior | Covering tests |
| --- | --- | --- |
| CMH-REVIEW-01 | Hovering or keyboard-focusing any section heading (`h1`-`h6`) inside `#commentRoot` reveals a `cm-skip` "Mark reviewed" badge just right of the title; activating it records a marker and renders a green `Reviewed` badge, and activating it again clears the marker. The badge/button are runtime-only and add no commentable text or offsets. | `tests/90-section-review.spec.js` - `marking a heading reviewed shows the badge and toggling clears it (CMH-REVIEW-01)` |
| CMH-REVIEW-02 | Review markers persist in `localStorage` under `<comment-key>::reviews` and restore after reload; the badge and TOC filter are `cm-skip` runtime chrome, so they never enter a Plain HTML export. | `tests/90-section-review.spec.js` - `a review marker survives reload and the chrome is absent from Plain export (CMH-REVIEW-02)` |
| CMH-REVIEW-03 | The section content hash covers the non-`cm-skip` text from the heading to the next same-or-higher heading and ignores transient DOM state (a highlight wrap, collapse state); editing the section's content flips a reviewed badge to amber `Changed - re-review`, while an unchanged reviewed section stays green after a reload. | `tests/90-section-review.spec.js` - `editing a reviewed section flips it to Changed while an unchanged one stays Reviewed (CMH-REVIEW-03)` |
| CMH-REVIEW-04 | An open (un-handled) comment anchored anywhere in a section puts it in the blue `Commented` state (an overlay with precedence over Reviewed/Changed/Unreviewed); deleting the comment reverts to the recomputed underlying state, a handled comment never changes the state, and clicking a Changed/Commented badge re-stamps the current hash (one-click re-review). | `tests/90-section-review.spec.js` - `an open comment marks the section Commented and one-click re-review re-stamps it (CMH-REVIEW-04)` |
| CMH-REVIEW-05 | The side-TOC carries a `cm-skip` segmented filter (All / Reviewed / Unreviewed / Commented / Changed) that collapses every section not in the chosen state and expands the rest; `All` re-expands every section; each TOC entry shows a per-state colored dot. | `tests/90-section-review.spec.js` - `the TOC review filter collapses non-matching sections and All re-expands them (CMH-REVIEW-05)` |
| CMH-REVIEW-06 | `Export as Portable` (both the inline embed and the NonPortable->standalone rebuild) and `Export Offline` bake the current markers into a dedicated `<script id="reviewedSections">` block (inside the EMBEDDED COMMENTS region, not in the Copy-all bundle) so they travel with the file, and `Export to Plain HTML` strips that region so the block is absent from a plain copy. | `tests/90-section-review.spec.js` - `Export as Portable bakes reviewedSections and Plain strips it (CMH-REVIEW-06)`, `Export Offline also bakes the review markers (CMH-REVIEW-06)` |
| CMH-REVIEW-07 | `tools/authoring/mark_reviewed.py` writes/updates/clears the `reviewedSections` block deterministically (computing each marker's hash with the shared `section_hash.py`, refusing unsafe ids/hashes, preserving the dominant newline, and inserting the block after the real embeddedComments block - decoy-safe, case-insensitive close), and `validate.py` accepts the block only as a JSON object whose every marker carries a safe base36 hash. | `tests/test_mark_reviewed.py`; `tests/test_validate_layer.py` - `test_reviewed_sections_block_is_validated` |
| CMH-REVIEW-08 | The JS runtime (`assets/js/84-section-review.js` `cmhSectionHash`) and the Python helper (`section_hash.py` `cmh_section_hash`) produce identical section hashes for a shared committed fixture set, and their DOM/HTML extractors agree over a real document - both exclude cm-skip chrome, script/style/template, and runtime-transformed blocks (diffs, KQL, mermaid, chart canvases, editable notes) so the hash covers the section's stable prose - so the two implementations cannot drift. | `tests/test_section_hash_golden.py`; `tests/90-section-review.spec.js` - `the runtime section hash matches the JS/Python golden (CMH-REVIEW-08)`, `the runtime and Python extractors agree on a doc with script/style/void and transformed blocks (CMH-REVIEW-08)` |
| CMH-REVIEW-09 | Section review reaches ANY existing document after `tools/authoring/upgrade.py` runs: because the runtime and its CSS live in the swapped CSS/COMMENT UI/JS regions and the runtime is localStorage-first and null-safe, a document produced before the feature existed (the real `dev/upgrade-corpus/v1.117.0.html` snapshot, which has no review chrome) gains a working review badge, mark-reviewed, and (once a section is marked reviewed) the side-TOC review filter after upgrade - with no baked `reviewedSections` block - and the marker persists across reload. | `tests/91-upgrade-review.spec.js` - `a pre-feature document gains a working section-review UI after upgrade.py (CMH-REVIEW-09)` |
| CMH-REVIEW-10 | The section-review chrome stays DORMANT until the reviewer starts: on a document with no review markers and no comments, no persistent state badge is shown (only the hover-only unreviewed "Mark reviewed" affordance) and the side-TOC review filter is hidden. It activates - the filter appears and per-entry status marks render - as soon as any section is marked reviewed OR the first comment is added (`window.__cmhReview.active()` reflects this). | `tests/90-section-review.spec.js` - `the review filter and TOC status marks stay dormant until the first review or comment (CMH-REVIEW-10)`, `adding the first comment activates the dormant review UI (CMH-REVIEW-10)` |
| CMH-REVIEW-11 | While active, each side-TOC entry carries a single-character status badge reflecting its section state - `R` reviewed (green), `C` commented (blue), `!` changed (amber), and a hollow badge for unreviewed - rendered as a CSS pseudo-element (`data-cmh-mark`) so the letter never enters the TOC link text that search and deep-links read; each non-unreviewed mark carries an accessible `role="img"` label. | `tests/90-section-review.spec.js` - `each TOC entry shows a single-character status badge reflecting its state (CMH-REVIEW-11)` |
| CMH-REVIEW-12 | The reviewed state travels with the file: marking a section reviewed then Export as Portable or Export Offline bakes the marker into the `reviewedSections` block, and reopening the exported file with no network loads the marker, activates the review UI (filter + TOC mark), and shows the reviewed badge. | `tests/90-section-review.spec.js` - `the reviewed state persists through Portable/Offline export and reopening activates the review UI (CMH-REVIEW-12)` |
| CMH-REVIEW-13 | Export bakes only markers whose heading still exists in the current document: a marker orphaned because its heading was removed is pruned from the `reviewedSections` block, so a shared Portable/Offline copy never leaks stale review metadata (headingText, reviewedAt, hash) for a section that no longer exists. | `tests/90-section-review.spec.js` - `export prunes orphan markers whose heading no longer exists (CMH-REVIEW-13)` |
| CMH-REVIEW-14 | Each side-TOC review-filter button carries a live per-state count rendered as an inline `(N)` beside its label (`All` shows the total section count; the four states partition it, so reviewed+unreviewed+commented+changed equals All); the count updates on every `refreshReviewUI` as sections are marked reviewed, commented, or changed. The count span is `aria-hidden` and the number is folded into each button's `aria-label` so it is announced once. The label and count stay on one line per button (`white-space: nowrap`) and the group wraps into rows without horizontal overflow even with two-digit counts. | `tests/90-section-review.spec.js` - `each review-filter button shows a live per-state count that stays on one line (CMH-REVIEW-14)` |
