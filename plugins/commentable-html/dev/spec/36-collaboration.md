## Collaboration: author attribution and comment threads

| Feature id | Behavior | Covering tests |
| --- | --- | --- |
| CMH-AUTHOR-01 | The reviewer's display name is a per-browser value stored in `localStorage` under `cmh::author` and seedable by the author via `data-cm-author` on `#commentRoot`. A sidebar identity control (`Commenting as`) shows the current name and lets the reader set or change it at any time; changing it applies to NEW comments only and never rewrites the author stamped on past comments. A new comment/reply is stamped with the current name at creation time (unset = unattributed). | `tests/92-threads.spec.js` - `the reviewer name is set via the identity control and stamps new comments only (CMH-AUTHOR-01)` |
| CMH-AUTHOR-02 | Every attributed comment and reply shows an author pill at the start of its note in the sidebar card, colored by a stable hash of the name (same name -> same color); an unattributed comment shows no pill. Author names are sanitized (control characters/newlines stripped) and capped at 60 characters at the load/merge choke point, so a hostile name cannot break out of the pill markup. | `tests/92-threads.spec.js` - `an attributed comment shows a hashed author pill and a hostile name is sanitized and capped (CMH-AUTHOR-02)` |
| CMH-AUTHOR-03 | Copy all labels each note with its author on the `Comment (by NAME):` / `Reply k (by NAME):` line, and the whole agent-facing bundle is hardened against structural-line injection: the author name AND every one-line metadata field (`Where`/`Section`/anchor labels/`Source`/image alt/etc.) fold ASCII newlines/tabs AND the Unicode line/paragraph separators U+0085/U+2028/U+2029 to spaces (so no attacker-authored value can forge a new logical line a `m`-flag regex or `splitlines()` consumer reads), the author's backtick/tilde runs are neutralized, and values are length-capped - so a hostile author name or metadata value can neither forge the untrusted-note fence nor inject a machine-trailer / HANDLED_IDS line. The Markdown export folds the same separators inside a note so a forged heading cannot escape the blockquote. | `tests/92-threads.spec.js` - `Copy all attributes each note and neutralizes a hostile author name (CMH-AUTHOR-03)` |
| CMH-THREAD-01 | A `Reply` action on a comment card opens a composer whose saved note becomes a reply in that thread; replies render stacked under the root card in chronological (oldest-first) order, each with its own edit and delete controls. The sidebar and toolbar counts show the number of THREADS (top-level comments), not the total note count, and the comment search's shown/total total counts threads. | `tests/92-threads.spec.js` - `replying adds a chronological reply under the root and the count stays per-thread (CMH-THREAD-01)` |
| CMH-THREAD-02 | Deleting a thread root removes the entire thread (the root and all its replies) and tombstones every id so the whole thread stays deleted across reload; deleting a reply removes only that reply and tombstones only its id, leaving the root and sibling replies intact. | `tests/92-threads.spec.js` - `deleting a root removes the whole thread while deleting a reply removes only it (CMH-THREAD-02)` |
| CMH-THREAD-03 | Copy all emits each thread as an initial `Comment` followed by clearly-labelled `Reply 1`, `Reply 2`, ... refinements (each note individually wrapped in the untrusted-note fence), an AGENT INSTRUCTIONS line explains threads, and the machine trailer's `HANDLED_IDS_JSON` array covers the root plus every reply id so a thread is marked handled and pruned together. | `tests/92-threads.spec.js` - `Copy all emits threads as comment-plus-refinements and handles all thread ids (CMH-THREAD-03)` |
| CMH-THREAD-04 | A reply is a single-level comment with a `parentId` (a SAFE_ID that differs from its own id) and no independent anchor; replies survive an Export as Portable / embedded-comments round-trip and reconstruct their threads on reload. A reply whose `parentId` does not resolve to a present thread root (root deleted, never embedded, or pointing at another reply) is an orphan and is pruned at load, never rendering, and stays pruned across reload; it is tombstoned only when its parent is genuinely absent from the embedded block (not merely crowded out by the merge cap), so a legitimate reply is not permanently lost. | `tests/92-threads.spec.js` - `threads survive an embedded round-trip and orphan replies are pruned (CMH-THREAD-04)`, `a live thread survives a real Export as Portable round-trip (CMH-THREAD-04)` |
| CMH-THREAD-05 | Saving a reply whose thread root was deleted while the reply composer was open is refused with a warning (rather than persisting a dangling orphan that would be silently pruned on reload), and deleting a reply, a whole thread, or all comments (Clear all) closes any open edit composer for a removed comment so an in-progress edit is not silently lost. | `tests/92-threads.spec.js` - `a reply is not saved once its root is gone and deleting closes open edit composers (CMH-THREAD-05)` |
| CMH-THREAD-06 | Editing a reply positions its composer at the thread root's anchor: because a reply has no anchor of its own, the composer resolves the root and dispatches on the root's anchor type (text mark, mermaid node, diff line, image, link, or widget), so a reply to a non-text root does not strand the composer at the viewport corner. | `tests/92-threads.spec.js` - `editing a reply positions the composer at the thread root anchor (CMH-THREAD-06)` |
