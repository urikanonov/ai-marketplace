## Collaboration: author attribution and comment threads

| Feature id | Behavior | Covering tests |
| --- | --- | --- |
| CMH-AUTHOR-01 | The reviewer's display name is a per-browser value stored in `localStorage` under `cmh::author` and seedable by the author via `data-cm-author` on `#commentRoot`. A sidebar identity control (`Commenting as`) shows the current name and lets the reader set or change it at any time; changing it applies to NEW comments only and never rewrites the author stamped on past comments. A new comment/reply is stamped with the current name at creation time (unset = unattributed). | `tests/92-threads.spec.js` - `the reviewer name is set via the identity control and stamps new comments only (CMH-AUTHOR-01)` |
| CMH-AUTHOR-02 | Every attributed comment and reply shows an author pill at the start of its note in the sidebar card, colored by a stable hash of the name (same name -> same color); an unattributed comment shows no pill. Author names are sanitized (control characters/newlines stripped) and capped at 60 characters at the load/merge choke point, so a hostile name cannot break out of the pill markup. | `tests/92-threads.spec.js` - `an attributed comment shows a hashed author pill and a hostile name is sanitized and capped (CMH-AUTHOR-02)` |
| CMH-AUTHOR-03 | Copy all labels each note with its author on the `Comment (by NAME):` / `Reply k (by NAME):` line, and the whole agent-facing bundle is hardened against structural-line injection: the author name AND every one-line metadata field (`Where`/`Section`/anchor labels/`Source`/image alt/etc.) fold ASCII newlines/tabs AND the Unicode line/paragraph separators U+0085/U+2028/U+2029 to spaces (so no attacker-authored value can forge a new logical line a `m`-flag regex or `splitlines()` consumer reads), the author's backtick/tilde runs are neutralized, and values are length-capped - so a hostile author name or metadata value can neither forge the untrusted-note fence nor inject a machine-trailer / HANDLED_IDS line. The Markdown export normalizes the same separators inside each note and keeps it inside its adaptive untrusted-data fence, so a forged heading cannot escape into the document. | `tests/92-threads.spec.js` - `Copy all attributes each note and neutralizes a hostile author name (CMH-AUTHOR-03)` |
| CMH-THREAD-01 | A `Reply` action on a comment card opens an EMPTY reply editor INLINE inside that thread card (Word-style, not a floating popup) whose saved note becomes a reply in that thread; replies render stacked under the root card in chronological (oldest-first) order, each with its own edit and delete controls. The sidebar and toolbar counts show the number of THREADS (top-level comments), not the total note count, and the comment search's shown/total total counts threads. | `tests/92-threads.spec.js` - `replying adds a chronological reply under the root and the count stays per-thread (CMH-THREAD-01)` |
| CMH-THREAD-02 | Deleting a thread root removes the entire thread (the root and all its replies) and tombstones every id so the whole thread stays deleted across reload; deleting a reply removes only that reply and tombstones only its id, leaving the root and sibling replies intact. | `tests/92-threads.spec.js` - `deleting a root removes the whole thread while deleting a reply removes only it (CMH-THREAD-02)` |
| CMH-THREAD-03 | Copy all emits each thread as an initial `Comment` followed by clearly-labelled `Reply 1`, `Reply 2`, ... refinements (each note individually wrapped in the untrusted-note fence), an AGENT INSTRUCTIONS line explains threads, and the machine trailer's `HANDLED_IDS_JSON` array covers the root plus every reply id so a thread is marked handled and pruned together. | `tests/92-threads.spec.js` - `Copy all emits threads as comment-plus-refinements and handles all thread ids (CMH-THREAD-03)` |
| CMH-THREAD-04 | A reply is a single-level comment with a `parentId` (a SAFE_ID that differs from its own id) and no independent anchor; replies survive an Export as Portable / embedded-comments round-trip and reconstruct their threads on reload. A reply whose `parentId` does not resolve to a present thread root (root deleted, never embedded, or pointing at another reply) is an orphan and is pruned at load, never rendering, and stays pruned across reload; it is tombstoned only when its parent is genuinely absent from the embedded block (not merely crowded out by the merge cap), so a legitimate reply is not permanently lost. | `tests/92-threads.spec.js` - `threads survive an embedded round-trip and orphan replies are pruned (CMH-THREAD-04)`, `a live thread survives a real Export as Portable round-trip (CMH-THREAD-04)` |
| CMH-THREAD-05 | Deleting a reply while its inline editor is open drops that editor with no orphaned draft, and Clear all still closes an open ROOT edit composer (top-level comment edits use the floating composer) so an in-progress root edit is not silently lost on bulk delete. | `tests/92-threads.spec.js` - `deleting a reply drops its open inline editor, and Clear all closes an open root edit composer (CMH-THREAD-05)` |
| CMH-THREAD-06 | The `Reply` action opens an EMPTY inline editor inside the thread card (a floating `.cm-composer` is NOT used) - it is never prepopulated with the text of the comment being replied to and shows no "reply to: ..." quote header. Cancel or Escape dismisses the editor without saving anything; Ctrl/Cmd+Enter (or Save) appends the reply to the thread in the sidebar. | `tests/92-threads.spec.js` - `Reply opens an EMPTY inline editor in the sidebar, never prepopulated with the comment text (CMH-THREAD-06)` |
| CMH-THREAD-07 | Editing an existing reply edits it IN PLACE in the sidebar (an inline editor, not a floating composer), prefilled with that reply's OWN current text; saving updates the reply and marks it edited. | `tests/92-threads.spec.js` - `editing a reply edits IN the sidebar, prefilled with that reply's own text (CMH-THREAD-07)` |
| CMH-THREAD-08 | Opening the first reply when no reviewer name is set reveals the sidebar identity editor so the reply can be attributed (a non-blocking prompt tracked separately from the first-comment nudge, so a reply is prompted even if an earlier comment composer already consumed the comment nudge); the reply still saves unattributed if the reviewer declines. | `tests/92-threads.spec.js` - `the first reply without a name prompts for a username, and still saves if declined (CMH-THREAD-08)` |
| CMH-THREAD-09 | An in-progress inline reply/edit draft is preserved across a sidebar re-render (comment sort, a note-typing debounce, a checklist change, etc.) - the editor is re-opened with the same text rather than silently dropped - and still saves normally afterward. | `tests/92-threads.spec.js` - `an in-progress inline reply draft survives a re-render such as sorting (CMH-THREAD-09)` |
