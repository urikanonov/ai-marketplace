## Deck integration (frontend-slides)

The commentable-html skill incorporates the third-party frontend-slides design system (MIT, Zara Zhang) as NATIVE CMH deck theme presets: curated preset profiles under `dev/skill/tools/deck/themes/` recolour the deck stage, components, and reusable recipe classes with system-font, contrast-safe tokens, so a themed deck needs no per-deck CSS and no per-deck hand-translation. A curated, pristine subset of the upstream skill is still vendored under `dev/skill/vendor/frontend-slides/` (enforced pristine by a required CI gate) as design provenance and the maintainer refresh source, and to supply the inlined `viewport-base.css` and the PPTX extractor. The author-time deck tools live under `dev/skill/tools/deck/`, and the runtime deck profile lives in the layer (activated only by `data-cmh-mode="deck"`, so non-deck documents are unaffected). See `references/deck-contract.md` for the runtime interface contract.

## Coverage gaps

The following features are documented but lack full automated behavior coverage, or have a known mismatch between spec and documentation:

**Remaining manual or design-only gaps:**
- CMH-DECK-01: deck-intent detection and the "confirm before building a real deck" gate are an agent-behavior convention in `SKILL.md` ("Deck capability"), not an automated test.
- CMH-DECK-03b: the "prefer the Anthropic `pptx` skill when installed, fall back to the local extractor" preference is prompt-level wording in `SKILL.md`; only the deterministic escaping and the local fallback (CMH-DECK-03a) are automated.

- CMH-CONTENT-06: prose cross-reference links are only partially covered by TOC anchor tests.
- CMH-KQL-01: a framed KQL figure (`figure.cmh-kql`) is now hard-enforced to carry a Run in Azure Data Explorer link (CMH-KQL-07, a validator error).
