Why commentable-html

AI agents increasingly answer with HTML, not markdown: plans, reports, design docs, code reviews. HTML is a spatial, richer medium - diffs and call-graphs as boxes and arrows, design tokens as swatches, data as real charts, long documents as collapsible, navigable sections. A wall of markdown flattens all of that.

But the moment you want to revise that artifact, the loop breaks down: you alt-tab between the rendered HTML and the AI chat, scroll to the spot you mean, and try to describe in prose what to change. The richer the document, the more painful the round-trip.

commentable-html closes that loop - it is a code review for your plan. It turns any standalone HTML into a self-contained review surface: select any paragraph, table cell, chart, diagram, or code block and comment in place, exactly where you mean it, the same way you would review a pull request. When you are done, one Copy all hands the agent a structured bundle of every note. You stay in the document and return to the chat only at the end - the loop gets tighter.

What you get

A full code-review layer baked into one portable HTML file.

Comment on anything

Prose, table cells, code blocks, KQL queries, charts, images, and mermaid diagram nodes are all selectable and commentable.

Single self-contained file

The CSS, UI, and runtime travel inside the document. Email it, archive it, or open it offline - it just works.

Round-trip to the agent

Copy all emits a Markdown bundle plus a machine-readable id list, so the agent acts on your notes and marks them handled.

Export as Portable

Embed the current comments into the file so a review travels with it. Share it and the recipient sees every note immediately.

Handled comments stay gone

Once the agent marks a comment handled, it is pruned on reload. The HTML file is the single source of truth, not the browser cache.

Deterministic tools

Ships stdlib-only Python tools and a strict validator so generated documents are correct by construction, not by luck.

How the review loop works

Feedback goes back to the agent as structured data, not pasted prose.

Self review

Generate

The agent produces a commentable HTML report.

Comment

You select text and leave inline notes in the sidebar.

Copy all

Hand the agent a Markdown bundle of every open comment.

Reload

The agent acts and marks ids handled; handled comments prune away.

Peer review

Self review first

Run the self-review loop to get the artifact in good shape.

Export as Portable

Download one self-contained copy and share it with a peer.

Peer comments

The peer opens it, adds inline comments, and Exports as Portable again - comments now embedded in the HTML.

Back to the agent

You open their Portable file and Copy all to feed their feedback to the agent.

Review someone's plan

Receive a plan

A colleague sends a plan or report as markdown or HTML.

Convert

Ask the agent to turn it into a commentable HTML with this skill.

Review inline

Add your comments in place, on exactly the parts you mean.

Export and send back

Export as Portable and return it with your comments embedded.

Two portability modes

Pick based on whether the document stays local or travels.

Non-portable default

References shared companion assets instead of inlining them, so each regeneration is far cheaper in tokens. Best for fast, personal iteration while you and the agent shape the document.

Portable

One self-contained file with the CSS, runtime, and comments inlined. Best when you want to share the document with a peer or keep it as a durable, long-term artifact - one click via Export as Portable.

Changelog

Changes to commentable-html. See the full changelog on GitHub.

[1.0.0] - 2026-07-10

Added
  • Offline, single-file commentable HTML review surface: reviewers select any paragraph, table cell, code block, KQL block, chart, image, or mermaid diagram and leave inline comments, then copy or export a bundle back to an agent, with no network dependency.
  • Portable mode (one self-contained file) and NonPortable mode (companion CSS/JS files).
  • Deterministic Python tooling (new_document, inline_images, chart_block, upgrade, validate, and more) plus a standard-library validator that enforces the structural invariants of a generated file.
  • Rich, commentable content: tables, Chart.js charts, mermaid diagrams, KQL blocks, code-review diffs, and inlined images.
  • Each generated file stamps the skill version that produced it in a <meta name="commentable-html-version"> in the head and in the visible footer.
Notes
  • The injected layer is version-agnostic: the region markers, the companion filenames, and the demo storage keys no longer embed a version. The single source of truth for the release version is dev/VERSION; build.py stamps it into the layer constant, plugin.json, the marketplace entry, and each document's version <meta>, and build.py --check guards against drift.
